
# 二、概要设计

## 2.1 总体架构设计

本系统采用**本地 B/S 架构（Browser/Server）**，在单机离线环境下运行。整体采用“**实时检测链路 + 语义增强链路**”的双通道设计，以兼顾实时性、可解释性与交互性。

* **实时检测链路**负责摄像头图像采集、YOLO目标检测、状态抽象、规则推理和实时状态输出；
* **语义增强链路**基于关键帧、行为状态摘要和规则结果，调用 Qwen2.5-VL-3B 进行语义解释和自然语言规则解析。

### 2.1.1 架构设计目标

1. **实时性优先**：学习行为检测与状态判定由轻量级 YOLO 模型和规则引擎承担，保证系统可持续实时运行。
2. **语义理解增强**：引入 Qwen2.5-VL-3B，对复杂或歧义场景进行语义解释，提高系统可解释性。
3. **可配置交互**：用户可以通过自然语言方式定义或调整行为规则，降低规则配置门槛。
4. **隐私优先**：所有图像、行为记录和模型推理过程均在本地完成，不依赖外部网络。

## 2.2 系统模块设计

系统由前端展示层、后端逻辑层、模型推理层、数据存储层四部分构成。

---

## 2.2.1 前端展示层（Vue3）

前端负责用户交互、状态展示、行为统计可视化以及规则配置操作。

| 模块      | 职责                       | 关键技术                |
| ------- | ------------------------ | ------------------- |
| 视频流展示模块 | 显示摄像头画面及检测框结果            | getUserMedia、Canvas |
| 状态展示模块  | 显示当前学习状态（专注/分心/离开/低效）    | Vue3 响应式            |
| 图表可视化模块 | 展示专注时长、分心次数、手机使用时长等      | ECharts             |
| 语义解释模块  | 展示 Qwen2.5-VL-3B 输出的状态解释 | Vue 组件              |
| 规则配置模块  | 输入自然语言规则，并展示解析结果         | Vue 表单、JSON 预览      |

### 前端设计重点

1. **主界面轻量化**：实时状态展示优先，避免复杂 UI 影响演示稳定性。
2. **信息分层呈现**：检测结果、规则判定结果、语义解释结果分别展示，避免信息混淆。
3. **规则可视化**：自然语言输入后，前端展示解析后的结构化规则，便于用户校验。

---

## 2.2.2 后端逻辑层（FastAPI）

后端作为系统核心调度中心，负责实时检测链路、语义增强链路及本地数据管理。

| 模块        | 职责                      | 关键技术                   |
| --------- | ----------------------- | ---------------------- |
| YOLO 推理服务 | 图像目标检测                  | YOLOv8n / ONNX Runtime |
| 状态抽象模块    | 将检测结果转为结构化状态            | Python                 |
| 规则推理引擎    | 基于规则判断行为状态              | Python 自定义规则系统         |
| 时间序列分析模块  | 滑动窗口分析行为连续性             | deque、NumPy            |
| VLM 语义服务  | 基于 Qwen2.5-VL-3B 生成状态解释 | Transformers / 本地推理框架  |
| 规则解析模块    | 将自然语言规则转换为结构化规则         | Qwen2.5-VL-3B          |
| 事件调度模块    | 控制何时触发 VLM 语义分析         | AsyncIO / 后台任务         |
| 数据管理模块    | 行为日志、规则配置、状态缓存          | SQLite                 |

## 2.3 核心技术设计

---

## 2.3.1 YOLO 目标检测模块

YOLO 模块承担系统的高频实时感知任务，用于检测学习场景中的关键对象。

### 模型选型

* 模型：YOLOv8n（或后续可替换为 YOLOv11n）
* 输入尺寸：640×640
* 推理方式：ONNX Runtime 本地推理

### 检测类别

* person
* cell phone
* laptop
* book

### 设计原则

1. 只承担**实时感知**任务，不承担复杂语义理解任务；
2. 保证高帧率和低延迟，作为整个系统的主链路基础；
3. 输出统一结构化检测结果，供状态抽象模块使用。

---

## 2.3.2 状态抽象模块

YOLO 输出的是低层视觉目标，系统需要将其转换为高层语义状态，以便规则系统使用。

### 状态抽象输出示例

```json
{
  "is_present": true,
  "using_phone": true,
  "using_laptop": true,
  "reading_book": false,
  "head_down": true,
  "stable_duration": 8
}
```

### 抽象逻辑说明

* `is_present`：是否检测到 person
* `using_phone`：手机是否与人体区域存在高关联
* `using_laptop`：是否存在电脑且处于学习工位区域
* `reading_book`：是否存在书籍并处于有效学习区域
* `stable_duration`：当前状态持续时长

### 模块意义

状态抽象层是系统从“视觉检测”走向“行为推理”的桥梁，也是项目区别于普通检测系统的关键设计。

---

## 2.3.3 规则推理引擎

规则引擎负责根据状态抽象结果和时间信息，输出实时学习状态。

### 规则示例

```python
if not is_present:
    state = "离开"

elif using_phone and stable_duration > 5:
    state = "分心"

elif using_laptop and not using_phone:
    state = "专注"

elif is_present and not using_laptop and not reading_book:
    state = "低效"
```

### 设计特点

1. **显式规则**：行为判定逻辑清晰、可解释；
2. **支持扩展**：后续可以新增更多状态与条件；
3. **支持动态更新**：可通过自然语言规则配置功能扩展。

---

## 2.3.4 时间序列分析模块

由于学习行为具有连续性，仅依赖单帧判断容易误判，因此需要引入时间维度分析。

### 方法设计

* 使用滑动窗口记录最近 5～10 秒状态；
* 对状态做平滑与持续时长统计；
* 仅当某种状态连续达到阈值时，才更新最终行为判断。

### 作用

* 降低 YOLO 瞬时误检带来的影响；
* 避免状态频繁跳变；
* 提高系统稳定性和可信度。

---

## 2.3.5 Qwen2.5-VL-3B 语义增强模块

为增强系统创新性与交互性，引入 Qwen2.5-VL-3B 作为本地语义理解模型，但不参与每帧实时检测。

### 模型角色定位

Qwen2.5-VL-3B 不是主检测器，而是**低频语义增强模块**，负责：

1. **状态解释生成**
   对规则推理结果进行语义化解释，说明系统为何做出当前判断。

2. **自然语言规则解析**
   将用户输入的自然语言规则转换为结构化规则配置。

### 调用策略

采用**事件触发式调用**，而非持续高频调用：

* 当规则结果存在歧义时触发；
* 当用户请求查看解释时触发；
* 当用户输入自然语言规则时触发。

### 输入信息

* 当前关键帧图像
* 最近一段时间的状态摘要
* 当前规则推理结果
* 用户输入的规则文本（规则配置场景）

### 输出形式

#### 场景1：状态解释

```json
{
  "state": "分心",
  "explanation": "系统检测到用户连续使用手机超过8秒，同时未检测到明显学习行为，因此判定为分心。"
}
```

#### 场景2：自然语言规则解析

```json
{
  "rule_name": "phone_distraction_10s",
  "condition": {
    "using_phone": true,
    "duration_sec": { ">": 10 }
  },
  "output_state": "分心"
}
```

### 模块价值

* 提升系统可解释性；
* 提升规则配置交互体验；
* 增强系统“智能体”特征，但不牺牲实时性。

## 2.4 数据库设计（SQLite）

系统采用 SQLite 作为本地数据库，存储行为记录、规则配置和语义解释日志。

### 2.4.1 行为日志表

```sql
CREATE TABLE behavior_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp DATETIME,
    state TEXT,
    duration INTEGER
);
```

### 2.4.2 规则配置表

```sql
CREATE TABLE behavior_rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_name TEXT,
    condition_json TEXT,
    output_state TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 2.4.3 语义解释日志表

```sql
CREATE TABLE semantic_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp DATETIME,
    state TEXT,
    explanation TEXT
);
```

## 2.5 接口设计

系统采用 RESTful 本地接口设计。

| 接口                  | 方法   | 描述              |
| ------------------- | ---- | --------------- |
| `/detect`           | POST | YOLO 目标检测       |
| `/state`            | GET  | 获取当前行为状态        |
| `/stats`            | GET  | 获取学习行为统计        |
| `/log`              | POST | 写入行为日志          |
| `/semantic/explain` | POST | 生成当前状态语义解释      |
| `/rules/parse`      | POST | 将自然语言规则解析为结构化规则 |
| `/rules/save`       | POST | 保存规则到本地规则库      |
| `/rules/list`       | GET  | 获取当前规则列表        |

## 2.6 部署设计（离线运行）

系统在本地单机环境部署，所有模型和依赖均打包随程序分发。

### 部署组成

* 前端：Vue3 + Vite 构建后的静态资源
* 后端：FastAPI 本地服务
* 检测模型：YOLOv8n ONNX 模型
* 语义模型：Qwen2.5-VL-3B 本地权重
* 数据存储：SQLite 本地数据库

### 启动流程

1. 用户双击程序启动后端服务；
2. 自动加载 YOLO 模型和 Qwen2.5-VL-3B 模型；
3. 浏览器自动打开本地页面；
4. 摄像头画面接入后开始实时检测；
5. 用户可查看状态、统计信息，并通过自然语言配置规则。

### 部署设计原则

* 完全离线运行
* 本地隐私保护
* 模块可替换、可裁剪
* 支持普通桌面设备演示

---

## 2.7 设计亮点总结

1. **双链路架构设计**
   将实时检测与语义增强解耦，兼顾实时性和可解释性。

2. **YOLO + 规则推理 + VLM 协同机制**
   YOLO 负责高频视觉感知，规则引擎负责实时判定，Qwen2.5-VL-3B 负责低频语义解释与规则配置。

3. **自然语言规则配置**
   用户无需手写代码或 JSON，可直接通过自然语言定义行为规则。

4. **隐私优先的本地智能系统**
   所有图像、状态和模型推理均在本地完成。

5. **工程实现可控**
   不把大模型强行塞进主链路，避免离线系统因模型过重导致不可用。
