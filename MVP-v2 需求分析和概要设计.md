
# 一、需求分析

## 1.1 项目背景与意义

在高校自习、编程学习、课程学习等场景中，学生普遍存在以下问题：

* 学习过程中容易受到环境干扰、注意力漂移、短时离席等因素影响，产生分心、低效、离开等行为；
* 学习状态缺乏客观量化反馈，用户往往只能凭主观感受判断自己的专注情况；
* 现有专注类工具大多以计时器或手动打卡为主，缺乏基于视觉感知的智能分析能力。

与此同时，现有基于摄像头的行为分析系统通常存在以下问题：

* **隐私风险**：学习环境图像与行为数据上传到云端，容易引发隐私泄露问题；
* **网络依赖**：断网环境下无法使用；
* **行为建模不合理**：部分系统默认摄像头可以稳定观测桌面物体，但在笔记本电脑前置摄像头场景下，桌面手机、书本、电脑本体通常不可稳定观测；
* **可解释性不足**：很多系统只能输出简单标签，无法说明判定依据。

基于上述问题，本项目拟设计并实现一个**完全离线运行的单前置摄像头学习姿态与分心行为分析系统**。系统以 YOLO（目标检测模型） 为高频感知核心，以规则引擎（规则推理模块）为实时决策核心，通过对上半身视觉特征进行状态抽象和时间序列（时序）分析，实现对学习状态的智能判定与反馈。

---

## 1.2 项目目标

本项目的总体目标是构建一个：

> **面向学习场景、支持本地离线部署、基于单前置摄像头实现实时姿态感知、状态抽象、规则推理与行为反馈的智能分析系统**

具体目标如下：

1. 基于 YOLO 实现上半身关键目标的实时检测；
2. 基于状态抽象与规则推理实现学习状态判定；
3. 基于滑动窗口（时间窗口）分析提升状态判断的稳定性和可信度；
4. 实现对“专注、分心、离开、低效”四类核心状态的实时输出；
5. 保证系统在本地离线环境下稳定运行，保护用户隐私；
6. 为后续引入更高层语义解释能力预留扩展接口，但本阶段不纳入主流程。

---

## 1.3 创新点概述

本项目的创新性主要体现在以下几个方面：

### 1.3.1 从目标检测升级为行为规则推理

传统视觉系统通常停留在“识别到了什么物体”这一层面，而本项目通过状态抽象与规则推理机制，将视觉检测结果进一步转化为学习行为状态，实现从“感知层”到“决策层”的提升。

### 1.3.2 面向单摄像头约束的行为建模

本项目明确承认笔记本电脑前置摄像头的观测边界，不依赖桌面手机、书本、电脑本体等不可稳定观测目标，而是基于用户上半身姿态、头部方向、在位情况等可观测特征进行分析，保证系统设计与真实部署条件一致。

### 1.3.3 引入时间序列分析提升判定稳定性

学习行为具有连续性，单帧图像无法准确表达用户状态。本项目通过滑动窗口记录状态变化，对行为进行持续性分析，从而降低瞬时误判。

### 1.3.4 隐私优先的离线智能架构

系统所有图像、日志和模型推理过程均在本地完成，无需联网，兼顾功能性与隐私保护。

### 1.3.5 工程实现可控

项目不依赖多摄像头和复杂外设，也不把大型模型强行塞入实时链路，而是在普通桌面设备上优先保证系统的稳定性、实时性和可演示性。

## 1.5 核心用例描述

### 用例 UC-01：实时学习状态分析

* **参与者**：学生用户
* **前置条件**：摄像头开启，系统正常运行

#### 基本流程

1. 用户启动系统；
2. 摄像头实时采集图像；
3. YOLO 检测上半身关键目标，如人、头部、手部；
4. 系统基于检测结果进行状态抽象；
5. 规则引擎根据状态与时间信息判定当前行为状态；
6. 前端实时展示当前状态，如专注、分心、离开、低效；
7. 系统将状态结果记录到本地日志。

#### 后置条件

* 用户可获得当前实时状态反馈；
* 行为状态被写入本地数据库。

---

### 用例 UC-02：学习行为统计查看

* **参与者**：学生用户
* **前置条件**：系统已运行一段时间并生成行为日志

#### 基本流程

1. 用户进入统计分析页面；
2. 系统读取本地行为日志；
3. 统计专注时长、分心次数、离开时长、低效时长等；
4. 以图表形式展示结果；
5. 用户查看当前学习行为分布。

#### 后置条件

* 用户获得学习行为的量化反馈。

---

### 用例 UC-03：规则参数调整

* **参与者**：学生用户或研究者
* **前置条件**：系统规则配置功能可用

#### 基本流程

1. 用户进入规则配置页面；
2. 查看当前规则阈值，如离开判定时长、分心判定时长等；
3. 修改阈值参数；
4. 系统保存更新后的规则配置；
5. 新规则立即参与后续行为判定。

#### 后置条件

* 用户完成规则参数配置；
* 系统规则获得更新。

## 1.6 非功能需求

### 1.6.1 性能需求

* YOLO 单帧推理延迟：**< 200ms**（CPU 环境）
* 实时处理帧率：**≥ 15 FPS**
* 状态更新延迟：**< 1 秒**
* 图表渲染延迟：**< 1 秒**
* 长时间运行下状态输出不得明显抖动或卡顿

---

### 1.6.2 可靠性需求

* 支持**长时间运行（≥ 2 小时）**
* 状态判定支持平滑与缓存，避免频繁跳变
* 低置信度检测结果应被过滤或降权处理
* 当部分辅助特征不可用时，系统主链路仍可独立运行
* 规则配置异常时，应回退到默认规则，不影响主链路运行

---

### 1.6.3 隐私与安全需求

* 所有图像与行为数据仅存储于本地
* 不允许将摄像头图像自动上传到外部服务器
* 用户可手动删除历史日志与配置记录
* 系统默认关闭联网依赖，保证离线运行

---

### 1.6.4 兼容性需求

* 操作系统：Windows / macOS / Linux
* 浏览器：Chrome / Edge / Firefox
* 摄像头支持：USB 摄像头 / 内置摄像头
* 支持普通桌面环境部署与演示

# 二、概要设计

## 2.1 总体架构设计

本系统采用**本地 B/S（浏览器/服务器）架构**，在单机离线环境下运行。整体采用“**实时检测链路**”的单通道设计，以保证系统结构清晰、部署简单、实时性稳定。

由于单前置摄像头无法稳定观测桌面物体，本系统不依赖书本、电脑本体等目标检测，而是基于用户上半身姿态进行行为分析。

### 2.1.1 架构设计目标

1. **实时性优先**：学习状态检测与判定由轻量级 YOLO 模型和规则引擎承担，保证系统可持续实时运行。
2. **观测真实**：仅使用单前置摄像头可稳定观测到的目标与姿态特征。
3. **可解释性**：规则判定逻辑清晰明确，便于调试和答辩说明。
4. **隐私优先**：所有图像、行为记录和模型推理过程均在本地完成，不依赖外部网络。
5. **可扩展性**：后续可在不破坏主链路的前提下接入更高层语义模块。

## 2.3 核心技术设计

---

## 2.3.1 YOLO 目标检测模块

YOLO 模块承担系统的高频实时感知任务，用于检测学习场景中的关键上半身目标。

### 模型选型

* 模型：YOLOv8n（后续可替换为更轻量或更适合姿态场景的版本）
* 输入尺寸：640×640
* 推理方式：ONNX Runtime 本地推理

### 检测类别

* person
* face 或 head
* hand（可选）

### 设计原则

1. 只承担**实时感知**任务，不承担复杂语义理解任务；
2. 保证高帧率和低延迟，作为整个系统的主链路基础；
3. 输出统一结构化检测结果，供状态抽象模块使用；
4. 不纳入桌面不可见目标，避免引入伪特征。

---

## 2.3.2 状态抽象模块

YOLO 输出的是低层视觉目标，系统需要将其转换为高层语义状态，以便规则系统使用。

### 状态抽象输出示例

```json
{
  "is_present": true,
  "face_visible": true,
  "head_down": false,
  "head_turned_away": true,
  "hand_near_face": true,
  "stable_duration": 6,
  "away_duration": 0,
  "inactive_duration": 4,
  "posture_stable": false
}
```

### 抽象逻辑说明

* `is_present`：是否检测到 person
* `face_visible`：是否检测到 face/head
* `head_down`：头部是否持续下垂
* `head_turned_away`：头部是否持续偏离屏幕方向
* `hand_near_face`：手部是否频繁进入面部附近区域
* `stable_duration`：当前姿态状态持续时长
* `away_duration`：离开持续时长
* `inactive_duration`：低活动持续时长
* `posture_stable`：是否持续保持相对稳定的学习姿态

### 模块意义

状态抽象层是系统从“视觉检测”走向“行为推理”的桥梁，也是项目区别于普通检测系统的关键设计。

---

## 2.3.3 规则推理引擎

规则引擎负责根据状态抽象结果和时间信息，输出实时学习状态。

### 规则示例

```python
if not is_present and away_duration > 3:
    state = "离开"

elif head_turned_away and stable_duration > 5:
    state = "分心"

elif is_present and inactive_duration > 10 and not posture_stable:
    state = "低效"

elif is_present and face_visible and not head_turned_away and posture_stable:
    state = "专注"
```

### 设计特点

1. **显式规则**：行为判定逻辑清晰、可解释；
2. **围绕可观测特征建模**：不依赖不可见桌面物体；
3. **支持扩展**：后续可以新增更多状态与条件；
4. **支持参数化配置**：阈值可通过配置调整。

---

## 2.3.4 时间序列分析模块

由于学习行为具有连续性，仅依赖单帧判断容易误判，因此需要引入时间维度分析。

### 方法设计

* 使用滑动窗口记录最近 5～10 秒状态；
* 对状态做平滑与持续时长统计；
* 仅当某种状态连续达到阈值时，才更新最终行为判断。

### 作用

* 降低 YOLO 瞬时误检带来的影响；
* 避免状态频繁跳变；
* 提高系统稳定性和可信度。

---

## 2.4 数据库设计（SQLite）

系统采用 SQLite 作为本地数据库，存储行为记录和规则配置。

### 2.4.1 行为日志表

```sql
CREATE TABLE behavior_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp DATETIME,
    state TEXT,
    duration INTEGER
);
```

### 2.4.2 规则配置表

```sql
CREATE TABLE behavior_rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_name TEXT,
    condition_json TEXT,
    output_state TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

| `/log`        | POST | 写入行为日志     |
| `/rules/save` | POST | 保存规则到本地规则库 |
| `/rules/list` | GET  | 获取当前规则列表   |

## 2.6 部署设计（离线运行）

系统在本地单机环境部署，所有模型和依赖均打包随程序分发。

### 部署组成

* 前端：Vue3 + Vite（前端构建工具）构建后的静态资源
* 后端：FastAPI 本地服务
* 检测模型：YOLOv8n ONNX 模型
* 数据存储：SQLite 本地数据库

### 启动流程

1. 用户双击程序启动后端服务；
2. 自动加载 YOLO 模型；
3. 浏览器自动打开本地页面；
4. 摄像头画面接入后开始实时检测；
5. 用户可查看状态、统计信息，并调整规则参数。

### 部署设计原则

* 完全离线运行
* 本地隐私保护
* 模块可替换、可裁剪
* 支持普通桌面设备演示

## 2.7 设计亮点总结

1. **单摄像头真实建模**
   系统严格围绕前置摄像头可观测特征设计，避免伪特征污染。

2. **YOLO + 状态抽象 + 规则推理协同机制**
   YOLO 负责高频视觉感知，状态抽象负责语义转换，规则引擎负责实时判定。

3. **时间序列稳定机制**
   通过滑动窗口与持续时长统计，降低瞬时误判和状态抖动。

4. **隐私优先的本地智能系统**
   所有图像、状态和推理均在本地完成。

5. **工程实现可控**
   不引入额外硬件，不依赖大型模型，优先保证系统可运行、可验证、可答辩。
