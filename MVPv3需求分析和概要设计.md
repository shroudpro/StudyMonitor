# StudyMonitor MVP v3 需求分析和概要设计

## 一、需求分析

### 1.1 项目背景与痛点回顾

在真实的学习场景（如高校自习、编程学习、课程学习）中，学生普遍存在注意力容易漂移、短时离席和低效学习（如长时间发呆、趴睡）等问题。传统的专注类工具多以简单的“倒计时机”或主观打卡为主，缺乏基于视觉感知的客观行为分析。

在前期版本（MVP v1/v2）的探索中，项目尝试了基于通用目标检测（YOLOv8）的场景观测路线，但暴露出了以下**核心痛点**：
1. **观测局限与误报率极高**：笔记本电脑的前置摄像头由于角度限制，根本无法稳定拍摄到桌面平放的手机、书本或电脑键盘，导致“使用手机”、“阅读书本”等特征充满噪音和伪命题。
2. **状态被相互吞噬**：简单的业务规则容易导致“低效”状态被强制包含进“专注”状态中（例如：一个人像雕塑一样盯着屏幕发呆，也会被系统一直认定为专注）。
3. **缺乏学习会话闭环**：工具仅作为一个“监控探头”在后台显示实时状态，但缺乏“开始学习”和“结束并结算分析”的完整业务流，数据失去了沉淀为统计报告的价值。

基于上述问题，**MVP v3 版本发生了根本性演进方案**：抛弃对不可见桌面物体的盲目预测，转向基于 **YOLO11-pose（人体姿态关键点）** 的视觉感知架构。严格基于人体的几何特征（面部、头部朝向、姿态位移）推演状态，引入完整的基于时间序列的“发呆（Inactive）”判断与 Study Session 数据结算闭环系统。

### 1.2 核心目标重塑
本系统（MVP v3）旨在构建一个 **完全离线本地部署、高隐私人身安全、通过单前置摄像头精准实现行为评价与完整数据统计闭环的智能学习分析系统。**

具体迭代目标：
1. **纯几何感知**：彻底剥离通用物体检测，仅依托 17 个人体关键点的确切拓扑关系来进行语义推理。
2. **静息发呆探测**：解决“看着屏幕但毫无操作（低效）”与“聚精会神”之间的混淆。
3. **强化状态分级**：理顺状态输出优先级（离开 > 分心 > 专注 > 低效（兜底兜盖））。
4. **数据生命周期**：支持一键开启新会话，重置参数以及落盘生成结构化的报告快照。

### 1.3 功能需求分解 (FM)

| 模块编号 | 功能名称 | 优先级 | 描述 |
| :--- | :--- | :---: | :--- |
| **FM-001** | 核心姿态追踪 | 必选 | 读取摄像头帧，使用 YOLO11n-pose 输出 17 个人体关键点坐标。 |
| **FM-002** | 几何抽象特征 | 必选 | 取代旧版的物品依赖，抽取出 `faceVisible`, `headDown`, `headTurnedAway`, `postureStable`, `inactiveDuration` 5 个纯生理特征。 |
| **FM-003** | 会话控制管理 | 必选 | 在前端包含【开始学习】、【重置】、【结束以分析】三大控制按钮及其绑定的业务周期。 |
| **FM-004** | 时序与规则引擎 | 必选 | 使用滑动窗口剔除单帧误检抖动，依照强逻辑优先级输出【专注/分心/离开/低效】四类基本态。 |
| **FM-005** | 日志与数据落地 | 可选 | Session 结束时汇总本次统计数据并持久化到本地 SQLite。 |
| **FM-006** | 结果统计与查阅 | 可选 | Session 结束后弹出概览表，显示：专注率、分心次数、各项累积时长。 |

### 1.4 非功能需求
* **性能基准**：基于 ONNX Runtime CPU 版运行，单阶段帧推理控制在 ~100-200ms 以内，不造成设备严重卡顿。
* **隐私保证**：网络与 API 调用均发生于 `127.0.0.1` 闭环，不发生外部传输。
* **抗干扰性**：后台能够长期（超过数小时）维护 Session 数据不丢失或崩溃，前端短线可依靠重连不影响后台时长累加。

## 二、概要设计

### 2.1 整体架构拓扑

### 2.2 核心模块详细设计

#### 2.2.1 YOLO11n-Pose 关键点抽取 (DetectorService)
系统剥离了原来包含 80 个大类的 COCO 识别方案，限定类别索引 `classes=[0]` (仅仅提取 Person)，并在 `DetectorService.detect` 获取以下重点拓扑：
- `0`: Nose (鼻子)
- `1/2`: Eyes (眼)
- `3/4`: Ears (耳)
- `5/6`: Shoulders (肩部)

该阶段摒除了各种干扰检测逻辑（手机不被包含在检测列表中），只负责将干净的 17 点列表发往下游。

#### 2.2.2 特征映射逻辑 (StateService)
摒弃了任何物理物品抽象，全面使用几何逻辑定义状态特征（`AbstractedState`）：

1. **`faceVisible`**：鼻子(Nose)拥有较高预测确信度且左右眼中至少一置信度可见。
2. **`headTurnedAway`**：一侧耳朵置信度极高，同时另一侧耳朵置信度极低，表明面部极度偏角。
3. **`headDown`**：鼻子距离肩膀水平线（由肩点连成中点计算垂直偏差）的距离短于常理阈值（视同下垂/趴倒）。
4. **`postureStable` 与 `inactiveDuration`**：
    * 建立一个容量固定的 Queue（约 15 帧长度），记录近期鼻子或肩膀中点的相对坐标轨迹。
    * 对轨迹数组计算**方差 (Variance)**。
    * 如果方差和 `< 0.01`，判定为 `postureStable` (姿态停顿稳定)。
    * 如果稳定连续存在，则让 `inactive_start_time` 递增。一旦 `inactiveDuration` (即一动不动的时间) 积攒超过安全阈值（如 5 秒），系统可以轻易认定用户虽然直视屏幕，但是处于发呆状态。

#### 2.2.3 状态决策引擎 (RuleEngine)
决策树重新排序，自上而下匹配满足的最高优先级特征，消除了旧版的歧义模糊：

1. **离开优先 (Away)**：如果 `isPresent=False` 且此状态维持超过 **3秒**，直接判定为【离开】。
2. **偏移优先 (Distracted)**：如果 `headTurnedAway` 为真，则认为分心（如果在偏移中持续超 5 秒置信度加深）。
3. **专注苛求 (Focus)**：必须同时满足四大条件才给与【专注】：
   - 人在现场 (`isPresent`)
   - 脸朝着相机可见 (`faceVisible`)
   - 未发生过度偏移转头 (`not headTurnedAway`)
   - **关键防发呆拦截**：姿态稳定，且绝对安静平稳时间还没超过临界值（`inactiveDuration < 5` 秒）。稍微动作一下会导致发呆缓存重置，保持正常的专注。
4. **低效兜底 (Low Efficiency)**：其余所有乱七八糟的状态一律打入低效。例如：
   - 处于专注姿势但僵硬长达 5 秒未曾移位（真死盯屏幕发呆）。
   - 长时间只漏出肩膀和耳朵在画面底端。
   - 身体姿态疯狂乱动方差爆炸。

#### 2.2.4 会话时间轴闭环 (Timeline & Session)
原系统一直启动但无尽头。为使其具有软件价值，新增 `StatsService.startSession()`, `resetSession()`, `stopSession()`。
* **业务状态机**：未 Start 时，只推送视频流、不进行 `duration` 增长和累计。
* **结束结算**: 当发起 Stop 请求时，立即冻结历史快照时间点，汇总期间内的所有状态百分比，自动调用 SessionLocal 落盘存储。

### 2.3 数据库重设计

为支持会话沉淀，引入核心产出表 **`study_session`**:
```sql
CREATE TABLE study_session (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    sessionId VARCHAR(50) UNIQUE NOT NULL,      -- 会话追踪码
    startTime DATETIME,                         -- 会话起始时间点
    endTime DATETIME,                           -- 会话结束时间点
    totalDuration INTEGER DEFAULT 0,            -- 监控涵盖总时长(秒)
    focusDuration INTEGER DEFAULT 0,            -- 专注占用时长
    distractedDuration INTEGER DEFAULT 0,       -- 分心占用时长
    lowEfficiencyDuration INTEGER DEFAULT 0,    -- 低效发呆兜底等问题时长
    awayDuration INTEGER DEFAULT 0,             -- 脱离座位时长
    distractedCount INTEGER DEFAULT 0           -- 分心动作触发了多少回
);
```
此结构可完整记录每次学习生命周期（无论多长），后期通过图表很容易做多天的学习专注率折线图渲染。

## 三、接口路由设计 (API & Socket)

MVP v3 依赖下述核心通讯网关：
1. **WS: `/ws/video`**: 以预设帧率发送处理后的视频帧 Base64 编码以及此时的 `AbstractedState`（含新的发呆计数 `inactiveDuration`属性等）以及框和线坐标。
2. **POST: `/api/session/start`**: 清空时间线队列和上一次的历史片段，启用 `isRunning` 并初始化发呆挂起计时戳和离开计时器。返回本次的 `sessionId`。
3. **POST: `/api/session/reset`**: 将当次记录无条件爆破，停止服务等待下一次启动。
4. **POST: `/api/session/stop`**: 关闭本 Session，自动计算所有比率和时长，返回详细的 JSON Report（并在内部落表）。

## 四、迭代复盘总结
MVP v3 的本质提升是：**从单纯的物件检测器，转变为了依靠几何变量与时序判定组合的行为模拟器，并完整接通了应用生命周期。** 
系统以很高的稳健性屏蔽了非人体设备的背景噪点，又以强硬的业务规则封堵了从前由于逻辑优先级错乱导致的各种不敏锐漏洞。如今这不仅是一个技术实验原型，更是一个具备了完整学习监督与报告能力的可用工具雏形。
